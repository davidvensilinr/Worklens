# AI Prompt — Tamper-Proof Employee Workspace Platform

---

## MASTER PROMPT

You are a senior full stack software architect. Build a complete, production-ready, tamper-proof employee workspace platform from scratch. This platform is a unified organizational ecosystem where every employee action is automatically tracked, timestamped, and stored as structured data. The platform is NOT an HR tool — it is a complete digital workspace where work actually happens, and every interaction becomes tamper-proof performance evidence.

---

## PRODUCT VISION

Build a platform called **"WorkProof"** (or suggest a better name) — a unified workspace where employees collaborate, communicate, and deliver work, while the system silently and automatically collects every measurable performance signal in the background. There are no manual surveys, no self-assessments, no manager ratings for performance data collection. The platform IS the proof. Every action is cryptographically timestamped and cannot be faked, backdated, or manipulated.

This platform feeds a separate ML pipeline (not part of this build) that predicts employee performance scores and salary hike recommendations. This platform's sole responsibility is to collect, store, and expose that data cleanly via API.

## CORE MODULES TO BUILD

### Module 1 — Authentication & Organization Setup
- Multi-tenant architecture (each organization is isolated)
- Role based access control: Super Admin, HR Admin, Manager, Employee
- Organization onboarding: company name, departments, teams
- Employee onboarding: profile creation, role assignment, team assignment, reporting manager assignment
- SSO support (Google, Microsoft)
- JWT based authentication with refresh tokens
- Every login/logout event timestamped and stored

### Module 2 — Document & File Workspace
- Upload PDF, DOC, DOCX, XLSX, images, and other file types
- Every file upload is timestamped (cannot be altered)
- Deadline tagging: every file/task can have a deadline attached
- Submission status: Pending → Submitted → Under Review → Approved / Rejected
- Version control: every edit creates a new version with timestamp and editor identity
- Multi-level verification workflow:
  - Employee submits → Manager reviews → HR approves
  - Each step timestamped independently
- File commenting and annotation system
- Rejection with reason — employee must resubmit
- Resubmission tracked separately (counts as revision)

**Data collected for ML:**
- `on_time_submission_rate` — % of files submitted before deadline
- `revision_request_rate` — % of submissions that were rejected and resubmitted
- `avg_resubmission_time` — how quickly employee corrects and resubmits
- `total_documents_submitted` — volume of work output
- `verification_pass_rate` — % of submissions approved on first attempt

### Module 3 — Task & Project Tracker
- Project creation with milestones and deadlines
- Task assignment: assigned by manager, accepted by employee
- Task priority levels: Critical, High, Medium, Low
- Task status: Assigned → In Progress → Submitted → Verified → Closed
- Dependency mapping: Task B cannot start until Task A is verified
- Progress percentage updated by employee, verified by manager
- Overdue detection: automatic flagging when deadline passes without completion
- Manager cannot close a task without verification step
- Employee cannot mark task complete without manager verification

**Data collected for ML:**
- `tasks_completed_on_time_rate` — % of tasks completed before deadline
- `projects_handled` — total projects participated in
- `overdue_task_count` — number of tasks missed deadline
- `task_complexity_score` — weighted average of task priorities completed
- `avg_task_completion_time` — efficiency metric

### Module 4 — Real Time Communication (Chat)

**Data collected for ML:**

- `mention_response_rate` — % of @mentions responded to within 1 hour

### Module 11 — Performance Data API (ML Feed)
This is the most critical module — it exposes all collected data to the external ML pipeline.

Build a secure REST API with the following endpoints:

```
GET  /api/v1/ml/employee/{emp_id}/performance-snapshot
     Returns all ML-ready features for one employee

GET  /api/v1/ml/organization/{org_id}/performance-snapshot
     Returns ML-ready features for all employees in org

GET  /api/v1/ml/employee/{emp_id}/history?months=12
     Returns time series of performance metrics

POST /api/v1/ml/appraisal/trigger
     Triggers data collection snapshot for appraisal cycle

GET  /api/v1/ml/employee/{emp_id}/audit-trail
     Returns full tamper-proof audit trail for one employee
```

**Note on satisfaction proxy metrics:**
Since there are no manual surveys, calculate satisfaction scores
automatically from behavioral signals:

- `environment_satisfaction_score`:
  Derived from: attendance rate + overtime willingness +
  escalation frequency (inverse) + sick days (inverse)

- `work_life_balance_score`:
  Derived from: overtime hours (inverse) + sick days (inverse) +
  late arrival rate (inverse) + leave utilization rate

- `job_involvement_score`:
  Derived from: response time (inverse) + meeting attendance +
  self initiated training + tasks completed rate

- `relationship_satisfaction_score`:
  Derived from: collaboration score + peer recognition given +
  mention response rate + communication frequency

- `job_satisfaction_score`:
  Derived from: recognition received + promotion recency +
  self initiated training + action items completed rate

All scores normalized to 1-5 scale to match ML model input format.

## ANTI-TAMPER & AUDIT ARCHITECTURE

Every single event in the platform must follow this pattern:

```
Event occurs (file upload, message sent, task completed)
        ↓
Event logged with:
→ Event ID (UUID)
→ Actor ID (who did it)
→ Timestamp (UTC, server side — never client side)
→ Event type
→ Payload (what happened)
→ SHA-256 hash of (Event ID + Actor ID + Timestamp + Payload)
        ↓
Stored in append-only audit log table
(no UPDATE or DELETE permissions on this table)
        ↓
Each record links to previous record hash
(blockchain-style chain — tampering one breaks the chain)
```

**Database rules for audit table:**
- No UPDATE allowed — ever
- No DELETE allowed — ever
- INSERT only
- Read access: HR Admin and Super Admin only
- Hash verification endpoint available for integrity checks

## IMPORTANT CONSTRAINTS

1. **Never build a manual survey or self-assessment form** — all satisfaction scores are derived algorithmically from behavioral data

2. **All timestamps must be server-side UTC** — never trust client-side timestamps

3. **The audit log table must have database-level constraints** preventing UPDATE and DELETE operations

4. **The ML API must return data in exactly the schema specified** in Module 11 — the external ML pipeline depends on this contract

5. **Multi-tenant isolation is non-negotiable** — one organization must never see another organization's data, even accidentally

6. **The platform collects data only** — it does NOT make any performance judgments, ratings, or hike recommendations. That is the external ML pipeline's job.

7. **All file uploads must be virus scanned** before storage

8. **GDPR compliance** — employees can request their data export, organization admins can delete employee data on offboarding (except audit log which is retained per legal requirements)

9. **Mobile responsive** — the platform must work on mobile browsers

10. **API versioning** — all APIs must be versioned (/api/v1/) to support future ML model updates without breaking changes

## FINAL NOTE TO THE AI BUILDING THIS

This platform's most critical output is the ML performance snapshot API (Module 11). Every other module exists to feed data into this API. When in doubt about what data to collect or store, ask: "Does this help calculate the performance metrics listed in the ML snapshot schema?" If yes — collect it. If no — it is optional.

The platform must never make performance judgments. It only collects evidence. The judgment happens in the external ML pipeline. Keep these two concerns completely separate.
